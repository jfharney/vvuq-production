eScore 2.0 Requirements and Test Methods

Req ID Requirement Test Case ID IN AN DEM EXE
R1 System shall use relational dataset(s) that can be summarized using graph constructs T1
R2 Calculate and persist a unique risk score for each entitiy (e.g. people, places, organizations) extracted from a historuical corpus of semi-structured and/or unstructured data T2
R3 Risk scores should aggregate across multiple user-specified criteria T3
R4 Allow user to differentially weight risk criteria T4
R5 Risk scores should be deterministic T5
R6 Risk scores should account for multi-hop traversals across underlying graph of interconnections between entities and keywords of interest T6
R7 Characterize or distinguish high-risk entities from lower risk entities (where risk is specified by proximity to various risk criteria of interest to a user) T7
R8 Risk criteria should be composed of multiple individual components (i.e., sets of keywords regarding a particular topic that might be of interest to a user such as “Threat Agents and Precursors”) T8
R9 For investigative purposes, a user should be informed on which particular risk criteria contributed to a given entity’s risk score T9
R10 Identification of domain-specific patterns of concern T10
R11 Both full pattern matches and partial matches are of interest T11
R12 Provide a summary of risk scores and pattern matches T12
R13 The summaries should be use-case specific and not rely on a single schema or ontology for every use case T13
R14 Risk scores should take into account the frequency of connection between entities as well as the distance of the connection T14
R15 System should be capable of updating the risk scores as new data becomes available T15
R16 System should be capable of allowing users to investigate results by pulling up raw data associated with results of interest T16
R17 System should leverage open source state-of-the-art distributed computing and graph analytics software so that there is a clear upgrade path as technology evolves and improves T17
R18 Approach must be generalizable and application-agnostic T18
R19 System should be capable of processing millions of documents, should be capable of operating on a distributed computing platform, and performance should scale with additional hardware (nodes and memory) T19
R20 The time eScore takes to answer user queries should be as close to real-time as possible T20
R21 Extract, transform, and load (ETL) and eScore algorithm processing speed for large datasets must be possible offline (e.g., overnight) T21
R22 User interactive sessions should not take more than a couple of minutes for moderate sized queries against indexed/loaded data T22
R23 System should produce the same results using the same input data and parameters on different computing systems, with the same hardware/software configurations T23
